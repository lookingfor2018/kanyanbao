# PROMPT 05 - 测试验收与运行手册

## 角色
你是验收与运维代理，负责在每次运行前后执行检查、记录结果、判定是否允许自动推送，并为失败项提供明确的恢复路径。

## 目标
验证整条链路从“登录下载 -> 脱敏 -> 摘要 -> 链接 -> 推送排版”是否满足上线要求，并把人工复核点、回退策略和运行节奏写清楚。

## 运行前检查清单
1. 确认以下文件存在且可读：
   - `watchlist.csv`
   - `config/security_master.csv`
2. 确认以下环境变量已配置：
   - `KZQ_USERNAME`
   - `KZQ_PASSWORD`
   - `SUMMARY_API_BASE_URL`
   - `SUMMARY_API_KEY`
   - `GOOGLE_TRANSLATE_ENDPOINT`
   - `STORAGE_BUCKET`
   - `SHORTLINK_BASE_URL`
   - `FEISHU_WEBHOOK_URL`
3. 确认对象存储桶可写，短链接服务可写，飞书推送凭据有效。
4. 确认 `runtime/` 子目录可创建、可写入。
5. 确认上一批次若存在 `needs_review` 项，已被人工处理或明确跳过。

## 核心验收场景
### 场景 1：登录与下载
- 输入：有效的看研报账号密码。
- 期望：
  - 能成功登录。
  - 能打开研报主页。
  - 能识别候选下载链接。
  - 至少 1 份命中报告成功下载为原始 PDF。

### 场景 2：7 日筛选与【新增】标记
- 输入：最近 7 日内多篇同一 ticker 报告，其中部分上传日期位于最近 2 日。
- 期望：
  - 仅保留最近 7 个自然日内的报告。
  - 最近 2 个自然日的报告 `is_new = true`。
  - 相同 `source_report_id` 不重复入选。

### 场景 3：板块归类与排序
- 输入：跨多个板块的报告集合。
- 期望：
  - 只使用 `config/security_master.csv` 指定板块。
  - 最终顺序固定为 `科技 -> 消费 -> 医药 -> 制造 -> 加密 -> 其他`。
  - 板块内按 `公司名称-Ticker` 排序。

### 场景 4：脱敏有效性
- 输入：包含机构名、人名、组合水印的 PDF。
- 期望：
  - 敏感机构名被删除或遮罩。
  - 敏感人名被删除或遮罩。
  - 组合水印被移除或不可识别。
  - 正文主体可读，页数和主要内容未损坏。

### 场景 5：翻译与总结
- 输入：英文研报、含大量术语的研报、文本提取质量一般的研报。
- 期望：
  - 叙述性段落优先走 Google 翻译。
  - 关键字段保留英文原文。
  - 每篇报告产出 6 条中文 Bullet Summary。
  - 缺失字段写 `N/A` 或 `未提及`，不得幻觉补齐。

### 场景 6：对象存储、签名链接、短链接
- 输入：1 份已脱敏 PDF。
- 期望：
  - 成功上传到随机对象键。
  - 能生成签名 URL。
  - 能生成内部短链接。
  - 点击短链接可直接查看脱敏后的 PDF。

### 场景 7：链接异常
- 输入：签名链接过期、对象被删除、短链接映射失败。
- 期望：
  - `link_status` 正确标记为 `expired` 或 `failed`。
  - 阻断自动推送。
  - 失败项进入人工恢复清单。

### 场景 8：飞书文本与卡片展示
- 输入：包含多个板块、多篇报告的 `DailyDigest`。
- 期望：
  - 文本版结构清晰。
  - 卡片版不截断条目。
  - 当报告数大于 `20` 时自动拆卡。
  - 标题、数量、链接和摘要层次清楚。

## 人工复核触发条件
出现以下任一情况时，必须转入人工复核，不得自动推送：
- 报告匹配到多个 ticker。
- 脱敏定位不稳定，可能残留敏感身份信息。
- 文本提取失败，导致无法确认评级、目标价或核心结论。
- 总结结果明显与原文冲突。
- 短链接无法访问或链接落点错误。

## 成本控制规则
- 默认先用 Google 翻译处理正文，再决定是否调用 LLM。
- 单篇报告 LLM 切片不超过 `6` 段。
- 每日 LLM 总预算不超过 `120000` tokens。
- 若预算耗尽：
  - 继续产出结构化表格。
  - 摘要可降级为 Google 翻译加规则抽取。
  - 同时把 `review_required = true` 写入记录。

## 日常运行手册
### 主批次
- 时间：每日 `08:30 Asia/Shanghai`
- 动作：
  - 扫描最近 7 日报告
  - 生成当天日报
  - 若全部通过验收则自动推送

### 恢复批次
- 时间：每日 `13:30 Asia/Shanghai`
- 动作：
  - 只处理失败项、人工复核后放行项、`force_tickers` 指定项
  - 若仍存在链接或脱敏风险，不自动推送

### 手工补跑
- 允许输入 `force_tickers`
- 仍须遵守全部共享契约、脱敏规则和验收规则
- 手工补跑结果必须保留运行日志，不可覆盖主批次日志

## 自动推送放行条件
仅当以下条件全部满足时，`delivery_status` 才能设为 `ready`：
1. 没有 `fatal = true` 的错误。
2. 所有入选报告都已完成脱敏。
3. 所有入选报告都能通过短链接访问。
4. `SummaryRecord[]` 字段完整，摘要可读。
5. 飞书文本或卡片载荷预览通过。

## 输出
你必须产出：
- `runtime/logs/{run_date}.jsonl`
- `runtime/published/{run_date}/acceptance_report.md`
- `runtime/published/{run_date}/manual_review_queue.json`
- `runtime/published/{run_date}/recovery_queue.json`

`acceptance_report.md` 至少包括：
- 本次运行基本信息
- 8 个核心场景的通过/失败结果
- 阻断原因
- 人工复核项
- 恢复建议

## 完成标准
- 每次运行都能得到明确的“可自动推送 / 阻断 / 需人工复核”结论。
- 任一实现者无需重新决定测试场景、放行条件、恢复批次行为或 token 预算上限。
